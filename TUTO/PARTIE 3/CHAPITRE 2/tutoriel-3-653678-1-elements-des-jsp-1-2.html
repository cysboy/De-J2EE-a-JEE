Eléments des JSP (1/2)

Le tutoriel que vous êtes en train de lire est en bêta-test. Son auteur souhaite que vous lui fassiez part de vos commentaires pour l'aider à l'améliorer avant sa publication officielle. Notez que le contenu n'a pas été validé par l'équipe éditoriale du Site du Zéro.
Par cysboy
Difficulté : Facile
Nous avons vu jusqu'alors que les pages JSP permettent d'insérer du code Java afin d'avoir un rendu dynamique de nos pages. Nous avons aussi abordé le fait que les JSP sont en fait des servlets où tout l'affichage est généré automatiquement.

Dans ce chapitre nous allons voir ce qu'on appelle vulgairement les éléments. Nous en avons déjà vu mais il est bon de refaire un point de la situation et de voir quelques nouveautés...

Présentation

Vous le savez déjà, les JSP sont des pages web avec du code Java leurs permettant d'avoir un rendu dynamique. A l'instar des autres langages offrant ce genre de fonctionnalités (PHP, .NET...) c'est côté serveur que le code sera interprété pour construire une page HTML en conséquence.
Pour ce faire, le serveur a besoin d'instructions mais surtout il doit savoir où commencent et où se terminent ces instructions. La technologie JSP, comme nous avons pu le voir, vous donne la possibilité, via des balises spécifiques, de donner des instructions au serveur comme :

  • importer une classe Java afin de s'en servir dans une JSP;
  • de définir une portion de code à exécuter;
  • ...


Ce qu'on appelle les "éléments", ce sont ces différentes directives que nous pouvons trouver dans une JSP. Nous allons maintenant voir que nous pouvons trouver 3 catégories d'éléments, à savoir :

  • les directives;
  • les scripts;
  • les actions.


Bon, je sens que vous êtes impatient...
Retour en haut

L'élément script

Comme vous vous en doutez sûrement, vous connaissez déjà cette catégorie d'éléments. En effet, nous nous en sommes déjà servi pour ajouter du code Java dans nos JSP mais, même si nous en avons déjà parler, je préfère revenir sur ce sujet afin que vous maîtrisiez bien ce point.

Les scripts sont donc des éléments qui vous donne la possibilité d'ajouter du code Java dans vos pages web afin de les dynamiser. il y a, au total, 3 type de scripts :

  • les scriptlets;
  • les déclarations;
  • les expressions.


Nous allons revenir sur ces points, dans l'ordre.

Les scriptlets


Les scriptlets sont des instructions contenant du code Java. il s'agit ni plus ni moins d'un bloc de code introduit dans votre page JSP. Ces blocs de code sont délimités par les balises suivantes : <% /* Bloc de code Java valide */ %>
Si vos souvenirs sont bons, c'est la première chose que nous avons vu lorsque je vous ai parlé des JSP...
Ainsi, si je reprend le code source de la page "index.jsp" du TP, la LoterieZ, vous pouvez tout de suite voir où sont les scriptlets :

Les déclarations


Comme son nom l'indique, une déclaration sert essentiellement à déclaré une variable (ou une méthode) afin de pouvoir l'utiliser dans votre JSP. Tout comme les scriptlets, nous avons déjà vu cette notion dans un précédent chapitre, vous devriez donc vous souvenir que, pour déclaré une variable d'instance dans une JSP, vous devrez utiliser les balises suivantes : <%! /* Bloc de code Java valide */ %>, sans oublier le "!".

Ceci aura donc pour effet de créer une variable d'instance lors de la transformation de la JSP en servlet (comme nous l'avons vu au chapitre précédent). De ce fait, cette variable sera accessible dans tous les scriptlets de la dite JSP. Il en va de même pour les méthodes ainsi créées. Voici un petit exemple afin d'étayer mes dires :

Les directives

Avant de commencer, je vous préviens, vous avez utilisé des directives...
Les directives permettent de préciser des informations dans la page JSP et, là aussi, il y a 3 types de directives différentes. Nous pouvons donc trouver les directives :
  • de page;
  • include;
  • d'actions

La particularité par rapport à des éléments de type script réside dans la balise utilisée, ici, nous devrons utiliser celle-ci :
Code : JSP - Sélectionner
1
<%@ /* directive */ %>


En fait, l'instruction pour invoquer les directives se construit comme ceci :

Code : JSP - Sélectionner
1
<%@ NomDeLadirective {attribut="valeur"} %>


Voici un exemple de directive :

Code : JSP - Sélectionner
1
<%@ page import="java.util.Date" %>


Lorsque vous importez une classe dans une JSP, vous utilisez une directive de type page...

La directive page


Vous connaissez déjà l'import de classe Java via cette directive mais cette directive vous permet de définir d'autres choses... Voici un tableau récapitulatif des valeurs possibles :

Attribut Description Valeurs possibles Valeurs par défaut Exemple
import Cet attribut vous permet d'importer une (ou plusieurs) classe(s) vous permettant ainsi de l'utiliser dans votre JSP. Une ou plusieurs classes Java ou un ou plusieurs package. Aucune. <%@ page import=" java.text.SimpleDateFormat, java.util.Date" %>
session Cet attribut vous permet de spécifier si la page en question peut accéder à la session en cours pour l'utilisateur. true ou false true. <%@ page session="false" %>
language Permet de spécifier quel langage utiliser pour traiter les instructions de la JSP. Il pourrait y avoir d'autres langages utilisés... java... java <%@ page language="java"%>
contentType Indique le type MIME de la page ainsi que l'encodage utilisé. Si le type de contenu est text/html, le contenu de la page sera interprété par le navigateur, se ne sera pas forcément le cas si vous indiquez qu'il s'agit de texte simple ou de XML... Nous reviendrons plus tard sur cette notion d'encodage. text/html;charset=ISO-8859-1
text/html;charset=UTF-8
text/plain;charset=ISO-8859-1
text/xml;charset=UTF-8
...		 <%@ page contentType="text/html; charset=ISO-8859-1"%> <%@ page contentType="text/xml; charset=UTF-8"%>
pageEncoding Défini l'encodage de la page : joue le même rôle que les balises <meta> HTML. Un encodage. Aucun. <%@ page pageEncoding="UTF-8" %>
info Offre la possibilité de retourner une description de la page. Une description quelconque. Dépend de la version de votre serveur d'application. <%@ page info="Informations sur la page en cours..."%>
errorPage Permet de spécifier une page d'erreur spécifique en cas d'exception non gérée. Une URL de page d'erreur. Aucune. <%@ page errorPage="/vue/error.jsp"%>
isErrorPage Indique si la page est une page d'erreur ou non. Dans le cas ou vous spécifiez cet attribut à true, vous pourrez accéder à un objet de façon implicite, sans l'importer : l'objet exception. true ou false false <%@ page isErrorPage="true" %>
isThreadSafe Indique si la page est Thread Safe ou non, c'est à dire que plusieurs utilisateurs peuvent y accéder en même temps. Dans le cas contraire, un seul client ne pourra accéder à la page. Ceci peut être utile si la dite page utilise une ressource non partageable entre plusieurs thread. true ou false true <%@ page isThreadSafe="false" %>
buffer Cet attribut permet de spécifier si le flot de sortie (la page générée) est placée dans un tampon avant l'envoi définitif. ce dernier s'utilise avec l'attribut autoflush. none ou 8kb ou <taille>kb 8kb <%@ page buffer="12kb" %>
autoflush Indique si le conteneur vide ou non le tampon lorsqu'il est plein. Honnêtement, vous ne devriez pas en avoir besoin... Idem pour l'attribut ci-dessus... true ou false true <%@ page autoflush="false" %>
extends Permet de définir la classe dont va hériter la servlet générée correspondant à la page JSP. Honnêtement, vous ne devriez pas en avoir besoin non plus... Une classe. Aucune. <%@ page extends="com.sdz.MyClass" %>
isELIgnored Indique à la JSP si les élements EL devront être ignorés ou non. Nous n'avons pas encore parler de cette notion mais ne vous en fait pas, c'est l'objet du prochain chapitre... true ou false true <%@ page isELIgnored="true" %>


Voilà, ça vous donne un beau panorama de ce que vous pouvez définir... Mais, comme je sais que vous êtes friand d'exemples, je vous propose de mettre quelques uns de ces attributs en pratique. Voici quelques JSP qui vous permettrons de voir ce que ces directives peuvent faire. Créez un nouveau projet et ajoutez-y ces JSP à la racine.

La directive include


Pour celle-ci, les anglophones l'auront deviné, elle sert à inclure. Mais inclure quoi ? une page JSP, dans une JSP !

Euh... Quel intérêt ?

Vous ne voyez pas ? Bon, imaginez-vous en train de développer une application J2EE qui devrait, au bas mot, faire une cinquantaine de pages JSP, avec logo, menu, pied de page et différents blocs qui se retrouvent dans toutes les pages... Ne me dite pas que vous allez mettre le code source du menu et des autres blocs commun dans toutes les pages ! Si demain votre chef vous demande de rajouter un point de menu, ou pire, de le refaire entièrement, vous allez devoir le refaire sur les 50 pages !

Ben pourquoi ? Le copier/coller ça existe !

Oui, mais le temps, c'est de l'argent ! Et même si le copier/coller existe, vous aller perdre un temps fou à repasser sur toutes les pages... et je vous fait grâce des erreurs de manipulation où, sur une page, vous allez sélectionner un balise en trop et votre copier/coller chamboulera toute la mise en page.

Bref, au lieu de modifier 50 fichiers, avec l'inclusion, vous pourrez n'en modifier qu'un et un seul ! Ce serait quand même beaucoup mieux. Cette directive se construit ainsi : <%@ include file="nomDeLaJSP.jsp"%>
En guise d'exemple, nous allons faire plusieurs pages avec un entête, un menu, un corps et un pied de page avec tous ces élements inclus, seul le corps changera...
