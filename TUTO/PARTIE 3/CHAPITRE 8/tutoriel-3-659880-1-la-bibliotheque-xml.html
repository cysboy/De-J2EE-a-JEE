La bibliothèque xml

Le tutoriel que vous êtes en train de lire est en bêta-test. Son auteur souhaite que vous lui fassiez part de vos commentaires pour l'aider à l'améliorer avant sa publication officielle. Notez que le contenu n'a pas été validé par l'équipe éditoriale du Site du Zéro.
Par cysboy
Difficulté : Intermédiaire
Comme je vous l'avais déjà dit précédemment, cette bibliothèque sert à travailler avec des flux XML.
Elle permet de parcourir et transformer un document XML.

Nous verrons aussi que pour parcourir le contenu d'un fichier XML, cette bibliothèque se base sur un standard : XPath, c'est donc ce par quoi nous allons commencer.

Ce chapitre risque d'être un peu complexe, alors accrochez-vous !

Rappel sur XML et Xpath

Cette bibliothèque permet de travailler avec des fichiers XML et se base sur le langage XPath.

Vous n'êtes pas sans savoir qu'un fichier XML est en fait une arborescence de balise qui pourrait être visualisées dans un JTree. Vous avez déjà dû voir ce genre de représentation... Rappelez-vous ! Au tout début du tutoriel, lorsque vous ouvriez le fichier web.xml avec l'éditeur XML qu'Eclipse utilise par défaut...

Afin de vous rafraîchir la mémoire, nous allons revoir ceci, avec un fichier XMl de test :

Bon, je vous l'accorde, l'utilité n'est que pédagogique... Mais l'idée est tout de même là.
Si vous ouvrez ce fichier avec l'éditeur XML d'Eclipse, vous devriez avoir quelque chose comme ça :



Chaque éléments d'un document XML s'appelle un noeud, ou node in english.
Chaque noeud peut avoir 0 ou 1 parent et 0 ou plusieurs enfants. Dans noter fichier d'exemple nous pouvons voir que le noeud "racine" n'a pas de parent mais possède deux noeuds enfants : les deux balises "tronc".

La représentation en arbre, comme vue ci-dessus, peux se lire ainsi :



Après cette brève introduction, vous devriez pouvoir faire l'analogie entre la façon dont les fichier XML sont construit et la façon de naviguer sur votre ordinateur... Vous ne trouvez pas que l'éditeur XML d'Eclipse affiche le fichier XML de la même façon que l'explorateur Windows ?



Partant de cette analogie, vous devez être familier avec ce genre de chemin : C:\Program Files\Apache Software Foundation\Tomcat 6.0\webapps.

Et bien le langage XPath revient à naviguer dans un fichier XML de la même façon donc, pour atteindre nos noeuds tronc, cela revient à écrire ceci : /racine/tronc.

La syntaxe XPath permet de spécifier les chemins à emprunter car ceux-ci peuvent être différents en fonction des cas mais aussi de récupérer plusieurs balises d'un seul coup. Dans l'exemple que je vous ai fourni à l'instant, l'instruction XPath /racine/tronc retournera tous les noeuds "tronc".

Voici quelques exemples de syntaxe XPath.

//feuille


Va récupérer tous les noeuds ayant pour nom "feuille" soit :



/racine/tronc/branche[1]


Va récupérer premier noeud qui a pour nom "branche" soit :



/racine/tronc/branche[1]/*


Va récupérer tous les éléments du premier noeud "branche" soit :



//@nom


Va récupérer tous les éléments qui ont pour attribut "nom" soit :



//branche[@nom]


Va récupérer tous les noeuds "branche" qui ont pour attribut "nom" soit :



//branche[@nom='branche2']


Va récupérer tous les noeuds "branche" qui ont la valeur "branche2" pour attribut "nom" soit :



Je ne vais pas vous montrer toutes les commandes XPath, il faudra un chapitre entier rien que pour ça...
Le but de ceci était de vous familiariser avec la syntaxe, vous voyez qu'il n'y a rien de difficile et que c'est très intuitif. Si toutefois vous voulez en savoir plus, je vous invite à lire ce tutoriel.

Nous sommes maintenant parés pour aborder la bibliothèque qui nous intéresse.

Utiliser un fichier XML avec JSTL

Nous allons maintenant voir quelles sont les actions de bases que nous pouvons faire avec cette bibliothèque mais, avant ça, il faut que je vous précise comment utiliser les flux XML avec JSTL...

Afin de pouvoir lire et parcourir un fichier XML via la bibliothèque, il va de soit qu'il nous faut ce dit fichier.
Pour y accéder, plusieurs façons de faire. Soit via le tag <c:import /> soit directement dans le corps des tag<x:... />.

<x:parse />, <x:out /> et <x:set />

<x:parse />


Ce tag permet d'analyser un document XML. C'est ce qui permettra à la plus part des autres tags XMl de pouvoir travailler avec le flux XML, c'est le point d'entrée.

Ce tag, comme vous vous en doutez, prend plusieurs attributs en paramètres :

Nom du paramètre Obligatoire ? Type du paramètre Description
var non. String Nom de la variable qui contiendra le flux XML analysé et utilisable par les autres tags.
scope non. String Nom du scope qui contiendra la variable définie par l'attribut "var".
varDom non. String Quasiment identique à l'attribut "var". La différence réside dans la façon de parser le document : nous n'en parlerons pas ici car ceci est très spécifique.
scopeDom non. String Utilisé lorsque l'attribut "varDom" l'est aussi. Ce couple remplace le couple "var" et "scope".
doc non. String ou Reader Source de la donnée XMl à analyser.
filter non. XMLFilter Défini le filtre à appliquer sur le document. Nous n'en parlerons ici.
systemId non. String L'identifiant système de l'emplacement du fichier (URI). Nous n'en parlerons pas non plus ici.
var non. String Nom de la variable qui contiendra le flux XML analysé et utilisable par les autres tags.

<x:out />


C'est via ce dernier que nous allons afficher les noeuds. Ce tag prend aussi des attributs en paramètre :


Nom du paramètre Obligatoire ? Type du paramètre Description

escapeXml non. String Tout comme pour la tag <:out />, permet de définir si les caractères spéciaux (<, >, &, ' et ") doivent être convertis en entité HTML.

Et le rendu :



La variable qui représente le flux XML, définie dans le tag <x:parse /> ne doit pas avoir d'accolades "{}" !
Ceci est la syntaxe correcte : <x:out select="$xml/racine/tronc[1]/branche[1]" />.
Alors que ceci est incorrect : <x:out select="${xml}/racine/tronc[1]/branche[1]" />.


Rien de compliqué ici, mis à part la syntaxe XPath...

<x:set />


Ce tag permet de définir une variable représentant un noeud, provenant d'une syntaxe XPath.


Nom du paramètre Obligatoire ? Type du paramètre Description
select oui. String Expression XPath à évaluer commençant par la variable définie par l'attribut "var" ou "varDom" du tag <x:parse />.
var oui. String Variable dans laquelle le résultat de l'expression XPath sera stocké.
scope non. String Nom du scope contenant la variable.

<%-- On défini une variable qui va contenir le premier noeud branche --%>
<x:set select="$xml/racine/tronc[1]/branche[1]" var="branche" />

	- a leurs propriétés.
 --%>

<p>Nom du premier fruit : <x:out select="$branche/fruit[1]/nom" /></p>
<p>Contenu de la dernière branche : <x:out select="$derniereBranche" /></p>




De la même façon que pour le tag <x:parse /> la variable créée avec le tag <x:set /> est utilisable dans le tag <x:out /> mais sans utiliser d'accolades !

<h1> Parsing du fichier en dehors d'import</h1>
<c:import url="/exemple.xml" var="docXml" />
<x:parse var="xml" doc="${docXml }" />

<x:if select="$xml/racine/tronc[1]/branche[1]/@nom='branche1'">
	<p>Le nom de la branche 1 est bien <x:out select="$xml/racine/tronc[1]/branche[1]/@nom" /></p> 
</x:if>

<x:choose>
	<x:when select="$xml/racine/tronc[1]/branche[1]/@nom='branche2'">

<x:transform>

Il vous est possible de transformer un fichier XML en autre chose (HTML par exemple), ceci grâce à une feuille de style XML : un fichier XSL.
Ce dernier va nous permettre la dite transformation en spécifiant comment traiter les données du fichier XML et comment les afficher.

Le but de ce chapitre étant de vous montrer comment utiliser la bibliothèque JSTL et non comment faire un fichier XSL, je vous invite à lire ce tutoriel afin de vous familiariser avec ce langage. Je partirais donc du principe que vous savez à peu près faire une feuille XSL... Vous verrez, ce n'est pas très compliqué et les balise XSL ressemble étrangement aux tag de la bibliothèque XML de JSTL (ou est-ce l'inverse... ), de plus, XSL repose aussi sur le langage XPath : vous ne serez donc pas trop dépaysé.


Pour illustrer le fonctionnement du tag <x:transform>, nous allons continuer d'utiliser notre fichier XML d'exemple et nous allons faire deux feuilles de styles différentes :
  • l'une affichera les données sous forme de tableau;
  • l'autre sous forme de liste.

Mais, avant tout, voici la liste des attributs de ce tag :

Nom du paramètre Obligatoire ? Type du paramètre Description
var non. String Nom de la variable qui contiendra le résultat de la transformation.
scope non. String Nom du scope qui contiendra la variable définie par l'attribut "var".
result non. Result Objet capturant le résultat de la transformation. Nous n'en parlerons pas ici.
doc non. String Source XML du fichier à transformer.
docSystemId non. String Identifiant système (URI) du fichier XML à transformer. Nous n'en parlerons pas non plus ici.
xslt non. String Ressource XSL servant à transformer le fichier XML passer dans l'attribut "doc".
xsltSystemId non. String L'identifiant système de l'emplacement du fichier (URI). Nous n'en parlerons pas non plus ici.
var non. String Nom de la variable qui contiendra le flux XML analysé et utilisable par les autres tags.


Vous avez aussi la possibilité de passer des paramètres à votre tag <x:transform>, via le tag <x:param>. Ce dernier est aussi le petit cousin de tag de la bibliothèque core et s'utilise dans le corps du tag <x:transform>.
