Difficulté : Intermédiaire
Dans le pays merveilleux de JSTL, il était une fois une bibliothèque vous permettant d'utiliser des méthodes. Sa cousine vous donnant la possibilité d'utiliser l'équivalent des mots clé du langage Java ne proposant pas ce genre de fonctionnalité, les développeurs de la plateforme ont donc palier à ce manque grâce à la bibliothèque fn.

Nous commencerons donc ce chapitre avec cette bibliothèque et nous finirons avec la création de vos propres tags JSTL, intéressant non ? Alors allons-y !

la bibliothèque fn

Comme je vous le disais dans l'introduction, cette bibliothèque vous permet d'utiliser des méthodes avec l'EL. Ces fonctions sont les équivalents de leurs homologues présentes dans le classe String : vous devriez donc connaître une bonne partie de celles-ci...

Par conséquent, je ne ferait qu'une brève présentation car, comme vous vous en doutez, ces méthodes sont très simples d'utilisations et utilisent les mêmes signatures que leurs consoeurs.

Petite précision toutefois : cette bibliothèque ne peut être utilisée qu'avec un conteneur JSP 2.0 car c'est une nouveauté de JSTL 1.1. Il vous faudra donc au minimum Tomcat 5.5 pour que cette bibliothèque fonctionne.


Voici donc les méthodes EL que vous pouvez utiliser, expliquée sous forme de code source.

${fn:contains()}


Cette méthode retourne un booléen en vérifiant si une sous chaîne de caractères est présente dans une autre :

Code : JSP - Sélectionner

<%@ taglib uri="http://java.sun.com/jsp/jstl/functions" prefix="fn" %>
<%@ taglib uri="http://java.sun.com/jsp/jstl/core" prefix="c" %>

<c:if test="${fn:contains(\"Bonjour les Zéros\", \"zéros\")} ">
	<p>La chaîne contient "zéros"</p>
</c:if>
<c:if test="${fn:contains('Bonjour les Zéros', 'Zéros') }">
	<p>La chaîne contient "Zéros"</p>
</c:if>


Vous devez mettre des "\" sinon la syntaxe des autres tags JSTL est cassé et vous aurez une erreur de compilation... Ou alors, utilisez des quotes simples "'"...



Ce code n'affiche que le contenu de la deuxième condition car cette méthode est sensible à la casse (minuscule et majuscule).

${fn:containsIgnoreCase()}


Fonctionne de la même manière que la précédente mais n'est pas sensible à la casse.

${fn:startsWith()} renvoi un booléen en vérifiant si une chaîne de caractère commence par une sous-chaîne et sa cousine vérifie si la chaîne se termine par une sous-chaîne.

${fn:toLowerCase()} transforme la chaîne en minuscule alors que ${fn:toUpperCase()} la transforme en majuscule.

${fn:substring()}, ${fn:substringAfter()} et ${fn:substringBefore()}


Ces trois méthodes retournent des sous-chaînes de caractères.

${fn:substring()} retourne une sous-chaîne en fonction d'une chaîne et de deux indices : l'indice de début et celui de fin.
${fn:substringAfter()} retourne la sous-chaîne de caractères après la sous-chaîne spécifiée.
${fn:substringBefore()} retourne la sous-chaîne de caractères avant la sous-chaîne spécifiée.

${fn:split()} et ${fn:join()}


${fn:split()} permet d'éclater une chaîne de caractères dans un tableau en fonction d'un délimiteur.
${fn:join()} permet de fusionner un tableau en une chaîne de caractère en utilisant le délimiteur comme séparateur de chaîne.

<c:set var="tab" value="${ fn:split(phrase, ' ') }" />

<c:forEach items="${tab}" var="substring">
	<p><c:out value="${substring }" /></p>
</c:forEach>

${fn:trim()}


Permet de supprimer les espace en début et fin de chaîne. Ainsi, ce code ${ fn:trim(" Bonjour les Zéros ") } retirera les trois espaces présent en début et en fin de la chaîne passée en paramètre.

${fn:length()}


Retourne la longueur d'une chaîne de caractères, donc ceci ${ fn:length("Bonjour les Zéros") } nous renverra 17.

${fn:indexOf()}


Nous retourne l'index de la sous-chaîne dans la chaîne donc ceci ${ fn:indexOf("Bonjour les Zéros", "les") } nous renverra la position de la première lettre de la sous-chaîne "les", à savoir 8.

${fn:escapeXml()}


Transforme les caractères XML en équivalent HTML. Ce code ${ fn:escapeXml("Bonjour & les <Zéros>") } retournera donc ceci : Bonjour &amp; les &lt;Zéros&gt;

Créez vos propres bibliothèques JSTL

Nous allons maintenant voir comment créer vos propres bibliothèques JSTL. Ceci peut s'avérer très utile dans la mesure où vous avez besoin de certains comportements ou fonctionnalités non présentes dans les bibliothèques de base.

Pour illustrer la création d'un nouveau tag, nous allons en développer un qui permet d'afficher le contenu d'une Map sous forme de tableau. Pour arriver à nos fins, nous allons devoir faire plusieurs choses :
  • créer un fichier .tld qui contient la définition de notre nouveau tag;
  • créer une classe java qui va contenir les actions à exécuter;
  • faire une JSP qui utilise ce nouveau tag.

Nous avons déjà fait, dans un précédent chapitre, un fichier .tld, vous ne serez donc pas dépaysé. Le fichier pour un tag JSTL est légèrement différents que pour une méthode EL, voici celui que je vous ai concocté pour notre exemple :

TableFromMap.tld

<taglib xmlns="http://java.sun.com/xml/ns/j2ee"
	xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
	xsi:schemaLocation="http://java.sun.com/xml/ns/j2ee/web-jsptaglibrary_2_0.xsd"
	version="2.0">

	<tlib-version>1.2</tlib-version>
	<short-name>MonTag</short-name>
	<uri>table</uri>

	<tag>
		<description>Génère un tableau depuis une Map</description>
		<name>table</name>
		<tag-class>com.sdz.jstl.TableFromMap</tag-class>
		<body-content>empty</body-content>
		<attribute>
			<name>map</name>
			<required>true</required>
			<rtexprvalue>true</rtexprvalue>
		</attribute>
		<attribute>
			<name>bgColor</name>
			<required>false</required>
			<rtexprvalue>true</rtexprvalue>
		</attribute>		
		<attribute>
			<name>color</name>
			<required>false</required>
			<rtexprvalue>true</rtexprvalue>
		</attribute>		
	</tag>
</taglib>


Je vous invite à créer ce fichier dans un dossier que vous nommerez "tlds" dans votre dossier WEB-INF.

Comme vous le savez, ce fichier sert au conteneur à définir la signature et le mode de fonctionnement de votre tag, il est donc primordial que le conteneur trouve et utilise ce fichier. Le fait de l'avoir mis dans le dossier que je vous ai spécifié n'est pas anodin... Pourquoi ? tout simplement parce que le conteneur va scruter plusieurs endroit de votre application en quête de fichier .tld.

Voici les endroits (étoiles rouge sur le schéma) ou vous pouvez déposer vos propres fichiers afin que le conteneur les interprète :



Vous pouvez ranger les TLD dans :
  • le dossier WEB-INF de votre application;
  • un dossiers "tlds" contenu dans le dossier WEB-INF;
  • le dossier META-INF des différents JAR que vous pouvez utiliser.
  • Dans n'importe quel sous-répertoire du dossier cité précédemment. Sur mon schéma il s'appelle "folder" mais son nom importe peu.

Donc, si vous rangez vos fichiers TLD à un de ces endroits, le conteneur les trouvera automatiquement.

Maintenant, attardons-nous un peu sur le contenu de ce fichier, il y a quelques variantes par rapport à celui que je vous avait fait créer pour la méthode EL.

L'entête XML est très importante :

<?xml version="1.0" encoding="ISO-8859-1" ?>
<taglib xmlns="http://java.sun.com/xml/ns/j2ee"
	xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
	xsi:schemaLocation="http://java.sun.com/xml/ns/j2ee/web-jsptaglibrary_2_0.xsd"
	version="2.0">


C'est elle qui va définir les balises autorisées, obligatoires... Vous n'avez pas à retenir par coeur cette définition, faites-vous un raccourcis Eclipse.

Nous avons ensuite la balise <tag> qui va définir la structure de votre tag personnalisé. Vous retrouvez :
  • <description> : une balise permettant de définir une description (c'est toujours utilise);
  • la balise <name> qui représente le nom qui sera utilisé dans votre tag : par exemple, dans notre cas, ça donnerait ceci <monprefix:table />;
  • la balise <tag-class> qui informe le conteneur sur la classe à utiliser pour traiter votre tag personnalisé;
  • la balise <body-content> qui permet de spécifier si votre balise comprend un corps ou non. Cette balise autorise plusieurs valeurs possibles :
    • empty : Votre tag ne devra pas avoir de corps;
    • scriptless : Votre tag peut avoir un corps mais il ne devra contenir que du texte, des expressions EL ou des actions standards ;
    • tagdependent : Votre tag ne devra contenir que du texte;
    • JSP : Votre tag pourra contenir tout ce que peut contenir une JSP.

  • et enfin, autant de balise <attribute> qu'il vous est nécessaire. Cette balise est relativement simple à comprendre, elle est composée de la balise :

    • <name> : défini le nom de l'attribut à utiliser dans le tag mais aussi dans l'objet Java qui déterminera le fonctionnement du tag;
    • <required> : indique si l'attribut est obligatoire ou non;
    • <rtexprvalue> : cette balise signifie runtime expression value et permet de spécifier si des expressions peuvent être utilisées pour définir l'attribut. Si cette balise est absente de votre définition, le conteneur considèrera que les expressions ne sont pas autorisées, ainsi, ce genre de chose ne sera pas autorisée : <monprefixe:table map="${maMap}" />.



Bon, nous avons maintenant noter fichier de description de tag, il ne nous reste plus qu'à créer la classe Java qui va traiter nos demandes et utiliser notre tag dans une JSP. C'est ce que nous allons faire tout de suite mais avant vous devez savoir quelques petites choses : cette classe est un peu spéciale, vous vous en serez doutez.

En effet, celle-ci devra hériter d'une classe qui s'appelle SimpleTagSupport, elle devra donc implémenter au moins une méthode, doTag(), qui est invoqué par le conteneur lorsque celui-ci cherche à utiliser un tag. Si votre tag a des attributs, etc'est le cas pour notre exemple, vous devrez aussi gérer des setters afin d'initialiser ces attributs dans la classe : ceux-ci devront respecter le nom de vos différents attributs, si vous avez un attribut "toto" dans votre TLD, il devra y avoir une méthode setToto() dans votre classe.

package com.sdz.jstl;

import java.io.IOException;
import java.util.Iterator;
import java.util.Map;

import javax.servlet.jsp.JspException;
import javax.servlet.jsp.JspWriter;
import javax.servlet.jsp.tagext.SimpleTagSupport;

public class TableFromMap extends SimpleTagSupport {

   private Map myMap;
   private String color = "white", bgColor = "black";
   
   /**
    * Cette méthode est appelée par le conteneur lorsqu'on
    * utilise le tag dans une page.
    * C'est dans celle-ci que nous allons parcourir notre Map
    */
   public void doTag() throws JspException, IOException{
      
      //On récupère l'objet qui va permettre d'écrire dans notre JSP
      JspWriter out = getJspContext().getOut();
      
      //Différents attribut CSS pour notre tableau
      String borderStyle = "border:1px solid black;";
      String collpase = "border-collapse:collapse;";
      String width = "width:600px;";
      String align = "text-align:center;";
      String styleColor = "color:";
      String styleBgColor = "background-color:";
      
      if(this.myMap != null && !this.myMap.isEmpty()){
         
         int nbre=0;
         
         //On initialise les couleurs de fond et de police
         styleColor += this.color + ";";
         styleBgColor += this.bgColor + ";";
         
         out.write("<table style=\"" + borderStyle + collpase + width + align + "\">");
         
         for (Iterator<Map<String, String>> i = this.myMap.values().iterator() ; i.hasNext() ;){
             Map<String, String> iter = null;
             
             //Vérification du type de Map passée en paramètre
             try {
               iter = (Map<String, String>)i.next();
            } catch (ClassCastException e) {
               out.write("<p>" + e.getMessage() + "</p>");
               // On sort de la boucle si le type d'objet n'est pas celui attendu
               break; 
            } 
             
             if(iter != null){             
               //pour la première lecture, on créer les entêtes
               if(nbre == 0){
                  out.write("<thead style=\"" + styleBgColor + styleColor + "\">");
                  for (Iterator<String> header = iter.keySet().iterator() ; header.hasNext() ; ){
                     out.write("<th style=\"" + borderStyle + "\">" + header.next() + "</th>");
                  }
                  out.write("</thead>");
                  out.write("<tbody>");
               }
               out.write("<tr>");
               //Maintenant, on remplit le tableau
               for (Iterator<String> content = iter.values().iterator() ; content.hasNext() ; ){
                  out.write("<td style=\"" + borderStyle + "\">" + content.next() + "</td>");
               }
               out.write("</tr>");
             }
            nbre++;
         }
         out.write("<tbody>");
         out.write("</table>");
      }
      else{
         out.write("<p>L'objet Map ne contient aucune information</p>");
      }      
   }
   
   //Ces accesseurs sont utilisés pour initialiser les attributs depuis la balise tag.
   //Vous remarquerez que les méthodes portent le même nom que les attributs dans le fichier TLD.
   public void setMap(Map<String, Map<String, String>> pMap){this.myMap = pMap;}
   public void setBgColor(String pBgColor){this.bgColor = pBgColor;}
   public void setColor(String pColor){this.color = pColor;}
   
}

package com.sdz.filter;

import java.io.IOException;
import java.util.SortedMap;
import java.util.TreeMap;

import javax.servlet.Filter;
import javax.servlet.FilterChain;
import javax.servlet.FilterConfig;
import javax.servlet.ServletException;
import javax.servlet.ServletRequest;
import javax.servlet.ServletResponse;

import com.sdz.model.TableHeader;

/**
 * Servlet Filter implementation class MapFilter
 */
public class MapFilter implements Filter {

   private FilterConfig config;

   public void destroy() {}

   public void doFilter(ServletRequest request, ServletResponse response, FilterChain chain) throws IOException, ServletException {
      
      String[] listAuteur = {"M@teo", "aerodark", "cysboy", "M@teo", "coyote"};
      String[] listCat = {"Site Web", "Infographie 3D", "Programmation", "Programmation", "Programmation"};
      String[] listTuto = {"HTML5 et CSS3", "Blender", "Java", "PHP", "JEE"};
      
      SortedMap<Integer, SortedMap> globalMap = new TreeMap<Integer, SortedMap>();
      
      for(int i = 0; i < listAuteur.length; i++){
         SortedMap<String, String> map1 = new TreeMap<String, String>();
         map1.put(TableHeader.PSEUDO.toString(), listAuteur[i]);
         map1.put(TableHeader.CATEGORIE.toString(), listCat[i]);
         map1.put(TableHeader.TUTO.toString(), listTuto[i]);
         globalMap.put(i, map1);
      }
      
      config.getServletContext().setAttribute("map", globalMap);      
      
      SortedMap<Integer, String> globalMap2 = new TreeMap<Integer, String>();
      globalMap2.put(1, "Attribut 1");
      globalMap2.put(2, "Attribut 2");
      globalMap2.put(3, "Attribut 3");
      
      config.getServletContext().setAttribute("map2", globalMap2);
      
      chain.doFilter(request, response);
   }

   public void init(FilterConfig fConfig) throws ServletException {
      this.config = fConfig;
   }
}

Créez des tags

Depuis la version 2.0 de la spécification JSP, vous avez la possibilité de créer des fichiers .tag qui peuvent être utilisé comme des tags JSTL. Ces fichiers ne sont ni plus ni moins que des JSP pré-développées que vous pouvez utiliser comme des tags et permettant de mieux gérer les inclusions de fichiers. Cela vous permettra d'avoir un tag disponible sans avoir recours à la lourde machinerie vu précédemment : ce n'est tout de même pas très simple d'écrire du HTML dans une classe Java, de plus, vous verrez que les fichier .tag sont particulièrement adaptés aux personnes n'ayant aucune connaissance particulière en Java (un designer par exemple).

Euh, d'accord... mais il existe déjà des directives et des tags qui permettent d'inclure des éléments dans des JSP ? Quel est l'intérêt de faire des fichiers tag ?

L'intérêt ? Vous allez vite le voir.
Dans beaucoup d'applications, il est courant de découper les pages en sous-sections (header, menu, contenu, footer...).

C'est déjà plus lisible, de plus, rappelez-vous bien que vous ne serez pas seul à travaillez sur votre projet, un designer pourrait avoir besoin de créer ce genre de chose de lui même, mais il y a un autre avantage à faire ainsi. Lorsque vous utilisez des directives, des actions ou des tags JSTL, les paramètres avec lesquels travaillent ces tags peuvent être stockés dans différents scopes : vous pouvez donc vous retrouver avec des paramètres dans une session, une requête ou autre alors que vous ne le souhaitiez pas de prime abord.

La façon de faire que je m'apprête à vous montrer utilise un nouveau scope, entièrement dédié aux tags. Les variables qui seront insérés dedans ne seront donc accessibles que dans nos tags et nulle part ailleurs !

On va encore être obligé de faire des TLD et tout le tintouin ?

Non... Cette façon de faire est plus simple, beaucoup plus simple. Il vous suffit de créer un fichier JSP contenant ce que vous voulez et l'enregistrer au format .tag dans ce dossier "/WEB-INF/tags".

Par exemple, j'ai récupéré le logo du SDZ, je l'ai enregistré dans un dossier "images" et j'ai créé un fichier "logo.tag" dans le dossier dont je vous parlais (WEB-INF/tags). Voici le contenu de ce fichier :

<%@ page language="java" contentType="text/html; charset=ISO-8859-1"

<html>

</body>

Et ce code nous affiche le logo du SDZ !
La seule différence avec la directive taglib habituelle réside dans l'utilisation de l'attribut "tagdir" en lieu et place de l'attribut "uri", sinon, rien de changé ! Alors, c'est tout de même très simple, non ?

Et comment tu définis le nom du tag à utiliser ?

Le nom du tag est son nom de fichier... Ici, c'est logo.tag donc la syntaxe est <tag:logo />, si il y avait d'autres fichiers, comme header.tag et footer.tag, vous pourriez les utiliser ainsi : <tag:header /> <tag:footer />

Et nous devons mettre nos fichiers tag dans ce seul dossier ?

Tout comme pour vos propres taglib, vous avez la possibilité de mettre vos tags à différents endroits, comme le montre le schéma ci-dessous :



Une subtilité demeure si vous déployez des tags dans des archives (JAR) : dans ce cas, vous devrez créer un fichier TLD décrivant le fonctionnement de votre tag ! Sans ça, le conteneur ne saura pas l'interpréter.

<%@ taglib uri="http://java.sun.com/jsp/jstl/core" prefix="c" %>

<%-- Définition des attributs que peut prendre le tag --%>
<%@ attribute name="title"  required="true" rtexprvalue="true" %>
<%@ attribute name="content"  required="false" rtexprvalue="true" %>

<%-- Affichage de l'attribut obligatoire "title" --%>
<h1><c:out value="${title }" /></h1>

<img src="images/sdz.png" alt="logo du SDZ" />

<%-- Affichage de l'attribut facultatif "content" --%>
<c:if test="${!(empty content) && (content ne null) }">
	<p> <c:out value="${content }" /> <p>
</c:if>

<%-- Gestion du corps du tag --%>
<%-- Ce qui sera mis dans le corp du tag sera récupéré par doBody et mis dans la variable "body" --%>
<jsp:doBody var="body"/>

<c:if test="${!(empty body) && (body ne null) }">
	<p>Corp du tag : <c:out value="${body }" /> </p>
</c:if>
