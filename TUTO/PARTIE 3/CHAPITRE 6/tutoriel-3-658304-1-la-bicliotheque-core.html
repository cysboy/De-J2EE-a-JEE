Présentation

Comme je vous le disais, la bibliothèque core permet d'utiliser des instructions de bases du langage Java ans vos JSP et ceci sans scriptlet...
Vous avez maintenant téléchargé et ajouter le .jar contenant la définition de toutes les méthodes que nous allons voir mais, pour pouvoir l'utiliser, vous devrez utiliser une directive dans vos JSP afin de faire le lien entre les classes et la JSP. Voici la directive à mettre dans vos JSP afin d'utiliser cette bibliothèque :

Code : JSP - Sélectionner
1
<%@ taglib uri="http://java.sun.com/jsp/jstl/core" prefix="c" %>


qu'est ce c'est que ce charabia ?


Comme je vous le disais dans le précédent chapitre, le lien entre les JSP et les méthode JSTL se fait via les fichiers .tld qui se trouve dans le répertoire WEB-INF de l'archive que vous avez téléchargé.
Cette directive permet de lier une ou plusieurs classes Java avec des éléments JSTL. Comment ? Tout simplement grâce aux fichiers .tld. voici comment le mapping se fait :



Ces informations sont donc obligatoires et ne sont pas laissées au choix du développeur... Voici donc le dernier pré-requis avant d'utiliser JSTL.

On parle communément de "tag" JSTL ceci parce que la traduction française de ce mot signifie "balise"

<c:out />

Ce tag permet d'afficher des données dans une page.
En fait, cela revient à faire <%= "toto" %> ou encore ${ "toto" }.

Comme beaucoup de tags, il peut prendre un ou plusieurs paramètres. Voici un tableau récapitulatif de ceux-ci :

Nom du paramètre Obligatoire ? Type du paramètre Description
value oui. String Donnée à afficher.
default non. String Donnée à afficher si la paramètre passé dans l'attribut "value" est null.
escapeXml non. String Détermine si les caractères spéciaux XML (<, >, &, ', ") doivent être convertis en caractère HTML. Par défaut, la valeur de cet attribut est "true"


Si ce tag revient à faire ${ "toto" }, quel est son utilité ?

Rien n'a attiré votre attention dans le tableau ci-dessus ? Pourtant le dernier attribut à une utilité, vitale !
Le fait de remplacer les caractères spéciaux XML en entité HTML permet de se prémunir contre certaine formes d'attaque WEB basées sur des injections de code javascript... Il me semblait vous l'avoir dit, il ne faut JAMAIS faire confiance aux données qui nous ont été fournies par les utilisateurs !

Je vais vous donner un exemple simple et vous verrez qu'avec ce tag, le code javascript ne sera pas interprété par le navigateur. Créez une JSP et mettez-y ceci :

Code : JSP - Sélectionner
1
${"<script>alert('Javascript exécuté ! Danger ! ');</script>" }


J'ai utilisé ici une simple chaîne de caractères, mais cette dernière aurait pu provenir d'une saisie utilisateur, dans un formulaire par exemple...


Lorsque vous allez sur votre page, vous pourrez voir qu'une boîte de dialogue s'affiche sous vos yeux... C'est mal, très mal !
Maintenant, renouvelons l'expérience mais, cette fois, nous allons utiliser le tag. Voici le code complet de notre JSP de test :

<c:set /> et <c:remove />

Ce tag permet d'initialiser des valeurs dans une Map ou un Bean ou, si la variable que nous tentons de modifier n'existe pas, de la créer. Voici un tableau des attributs de ce tag :

Nom du paramètre Obligatoire ? Type du paramètre Description
var non. String Nom de l'attribut à initialiser.
value non. String Valeur ou expression initialisant la variable passée dans l'attribut "var".
scope non. String Nom du scope qui contiendra la variable définie dans l'attribut "var". les valeurs possibles sont page, request, session ou application. Si cet attribut n'est pas renseigné, la valeur par défaut est "page".
property non. String Nom de la propriété où de la clé (s'il s'agit d'une Map) qui sera modifiée.
target non. Object Objet dont la propriété, définie par l'attribut "property" sera modifié. Il s'agira soit d'un Bean soit d'une Map.

package com.sdz.jstl.set;

import java.io.IOException;

import javax.servlet.ServletContext;
import javax.servlet.ServletException;
import javax.servlet.http.HttpServlet;
import javax.servlet.http.HttpServletRequest;
import javax.servlet.http.HttpServletResponse;
import javax.servlet.http.HttpSession;

public class Set extends HttpServlet {
	protected void doGet(HttpServletRequest request, HttpServletResponse response) throws ServletException, IOException {
		
		//pour le scope session
		HttpSession session = request.getSession();
		session.setAttribute("sessionAttribute", "valeur de l'attribut de session");
		
		//Pour le scope application
		ServletContext context = getServletContext();
		context.setAttribute("applicationAttribute", "valeur de l'attribut d'application");
		
		//Pour le scope request
		request.setAttribute("requestAttribute", "valeur de l'attribut de requête");
		
		Personne personne = new Personne();
		personne.setNom("Herby");
		personne.setPrenom("Cyrille");
		request.setAttribute("personne", personne);
		
		request.getRequestDispatcher("set.jsp").forward(request, response);		
	}
}

Rappelez-vous bien que ce qui se trouve dans le corps du tag est évalué avant affectation !

<c:if />

Voici donc les attributs de ce tag :

Nom du paramètre Obligatoire ? Type du paramètre Description

<c:url>, <c:import>, <c:redirect> et <c:param>

Nous allons faire un tire groupé car ce sont quatre tags relativement simples...
Le point commun entre tous ces tags est qu'ils travaillent tous pour créer des URL. Ils permettent de faire des liens HTTP vers des ressources internes à votre application ou à une ressource externe.

Par contre, si vous n'êtes pas familier du développement WEB, vous devez savoir une chose importante : les URL doivent être encodées afin d'être interprétables, ceci est une règle mondiale !

<c:param />


Commençons par le tag <c:param>. Celui-ci sert à passer des paramètres autres trois autres tags et possède deux attributs :

Nom du paramètre Obligatoire ? Type du paramètre Description
name oui. String Détermine le nom que prendra notre paramètre.
value non. String Indique la valeur du paramètre.


Ce tag s'utilise comme ceci <c:param name=”nom” value=”${nom}” />.

Comme je vous le disais, les URL doivent respecter certaines règles de construction et, pour ce faire, les caractères spéciaux DOIVENT être transformés pour être utilisables dans une URL. En effet, certains caractères comme :
  • l'espace;
  • les caractères accentués;
  • les caractères spéciaux;
  • ...


doivent être transformés pour pouvoir être récupérés et ne pas perdre d'informations, mais surtout ne pas corrompre l'URL, faute de quoi, l'URL ne sera pas utilisable et la ressource vers laquelle elle pointe ne sera pas accessible.

En bref, voici un exemple :


Code : JSP - Sélectionner
1
2
3
4
<c:param name="texte" value="Un texte avec espace" />
<!-- Nous donne ceci : texte=Un+texte+avec+espace -->
<c:param name="special" value="c@r@ctère spéciaux #!;?" />
<!-- Nous donne ceci : special=c%40r%40ct%e8re+sp%e9ciaux+%23!%3b%3f  -->


Et, vu que ce n'est utilisable que pour des URL, le fait de mettre ces deux paramètres nous donnerai, pour l'URL http://toto.fr/page.jsp :

http://toto.fr/page.jsp?texte=Un+texte+avec+espace&special=c%40r%40ct%e8re+sp%e9ciaux+%23!%3b%3f

Vous pouvez constater que les nom de nos paramètres sont bien ceux spécifiés, que leurs valeurs ont été encodés afin que les caractères spéciaux soit interprétables et que ces deux paramètres sont concaténés via le caractère : "&".

Voilà ce que fait ce tag : il sécurise les informations passées dans les URL.

Bon, voyons maintenant le tag suivant.

<c:import />



Contrairement à ce que son nom pourrait vous laisser penser, cette balise ne sert pas à faire des imports de classes Java dans une JSP. Elle sert à faire un include, tout comme la directive et l'action <jsp:include>. C'est donc une troisième méthode pour faire de l'inclusion de fichier. Voyons les attributs de ce tag :

Nom du paramètre Obligatoire ? Type du paramètre Description
url oui. String URL de la ressource à inclure dans la page.
var non. String Nom de la variable de scope qui contiendra la ressource à inclure sous forme de chaîne de caractères.
scope non. String Nom du scope qui contiendra la variable, vous connaissez maintenant.
varReader non. String Idem que l'attribut "var", mais contiendra la ressource sous forme d'un objet Reader au lieu d'une chaîne de caractères. Si cet attribut est utilisé, vous devrez utiliser le corps de la balise car cette variable n'est utilisable que dans celui-ci et, dans ce cas, il n'est pas possible d'utiliser le tag <c:param /> car l'URL doit être complète avant d'entrer dans le corps du tag. Ceci permet, par exemple, de travailler avec des flux XML à parser ou d'autres choses encore...
charEncoding non. String Spécifie l'encodage de caractères à utiliser dans la ressource. S'il n'est pas renseigné, c'est l'encodage de la ressource qui sera utilisé.
context non. String Spécifie le chemin du contexte à utiliser lors de l'inclusion d'une ressource locale. De ce fait, cet attribut commence OBLIGATOIREMENT par un "/". S'il n'est pas renseigné, cet attribut prend la valeur retournée par la méthode request.getContextPath().


Si les attributs "var" et "varReader" ne sont pas utilisés, la ressource sera directement ajoutée dans la page.


Et pourquoi utiliserait-on ce tag ?

Parce que ce tag permet une chose que les deux autres instructions ne permettent pas : inclure une ressource extérieure à votre application, comme une image hébergée sur un site particulier.

import javax.servlet.http.HttpServlet;

import javax.servlet.http.HttpServletResponse;

public class Foreach extends HttpServlet {
	protected void doGet(HttpServletRequest request, HttpServletResponse response) throws ServletException, IOException {
		
		//Un tableau tout simple
		String[] tabString = {"M@teo", "cysboy", "thunderseb", "Nesquik69", "Nanoc", "aerodark"};
		
		//Une liste d'entier
		List<Integer> listInteger = new ArrayList<Integer>();

		listInteger.add(1);
		listInteger.add(2);

		mapString1.put("3", "Value1-3");
		mapString1.put("4", "Value1-4");
				
		SortedMap<String, Integer> mapString2 = new TreeMap<String, Integer>();
		mapString2.put("1", 10);
		mapString2.put("2", 100);
		mapString2.put("3", 1000);
		mapString2.put("4", 10000);
		mapString2.put("5", 100000);

		SortedMap<String, Boolean> mapString3 = new TreeMap<String, Boolean>();
		mapString3.put("1", true);
		mapString3.put("2", false);

<c:forTokens />



Nous allons terminer ce chapitre sur ce tag qui n'est pas très différent du précédent, il permet juste de découper une chaîne de caractères en morceaux.

Nom du paramètre Obligatoire ? Type du paramètre Description
items oui. String Chaîne de caractères qui sera découpée grâce à l'attribut delims.
delims oui. String Liste de caractères qui vont permettre de découper la chaîne en morceaux.
begin non. int Si cet attribut est utilisé, l'itération commencera à l'index spécifié. Pour rappel, le premier éléments d'une itération est à l'indice 0 !
end non. int Permet de définir à quel indice (inclus) la boucle doit s'arrêter.
step non. int Permet de définir comment la boucle va itérer. Par défaut, la valeur est 1, donc avance de 1 en 1.
var non. String Nom de la variable qui contiendra le résultat de l'itération. Peut être une chaîne ou une implémentation de même type que pour l'attribut items dans le cas de tableau de tableau par exemple.
varStatus non. String Objet de type javax.servlet.jsp.jstl.core.LoopTagStatus permettant de savoir où en est l'itération.
